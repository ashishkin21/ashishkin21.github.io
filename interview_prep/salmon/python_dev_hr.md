### 3. "Что такое MCP? Есть ли у вас опыт с ним?" (ПРОБЕЛ)

MCP стандартизирует, как LLM-агенты обнаруживают и вызывают инструменты — по сути это контракт в стиле OpenAPI между агентами и backend-сервисами, с фокусом на описания инструментов, схемы входных данных и структурированные ответы об ошибках. В моём текущем проекте я построил эквивалентный слой без MCP: систему на основе вебхуков, где агенты вызывают FastAPI-сервис, который маршрутизирует к 11+ клиентам внутренних API (Pyrus, Confluence, KHub, LLM, Keycloak, Kafka и т.д.) с логикой повторных попыток, управлением авторизации и структурированными ответами. Принципы проектирования идентичны — понятные схемы, идемпотентные операции, осмысленные ошибки. Я уверен, что смогу быстро освоить MCP, поскольку я уже решил те же задачи, которые он решает.

> Мои текущие пользователи — буквально другие инженеры и их системы. В summarizer-it я спроектировал вебхук-хендлер так, чтобы он принимал универсальную структуру событий, а затем маршрутизировал внутренне на основе типа формы и шага — так вызывающему не нужно знать о выборе промптов, агрегации контекста или маршрутизации LLM. Я использую Pydantic-модели для строгой валидации входных данных, возвращаю структурированный JSON с едиными полями ошибок, и я построил класс `NonASCIIJSONResponse` для корректной обработки многоязычного вывода. Я также спроектировал логику повторных попыток и резервирования (например, повышение приоритета при 429, откат от qwen3-30B к qwen3-80B), чтобы вызывающие получали надёжные ответы без управления причудами LLM. Ключевой принцип: скрывай сложность, показывай чистые контракты.

### 9. "Опишите архитектуру вашего основного проекта"

> summarizer-it — это FastAPI-сервис с 3 эндпоинтами: вебхук-хендлер для событий Pyrus, эндпоинт BotRate-агента и health check. Основной поток: приходит вебхук → мы ищем подходящие промпты в реестре → собираем контекст параллельно из Confluence, KHub, Minerva, Graylog → вызываем LLM → отправляем результаты обратно в Pyrus, по почте или в VK Teams. Для многошаговых процессов есть цикл промптов, который переоценивает состояние задачи после каждого выполнения промпта. Я построил двух LangGraph-агентов: Prisma (параллельная агрегация контекста с опциональной фильтрацией по релевантности) и BotRate (параллельные запросы к KHub через Send API). Сервис работает в Docker/K8s с read-only root filesystem, секреты из Vault, метрики через Kafka и структурированное JSON-логирование.

### 10. "Как вы справляетесь с ошибками и повторными попытками?"

> Я построил собственный базовый класс `AsyncHTTPClient` с настраиваемой логикой повторных попыток — 5 попыток с 7-секундными интервалами. Для LLM-Integration при 429 клиент автоматически повышает приоритет запроса (low → normal → high). Когда попытки исчерпаны на основной модели, происходит откат к более крупной модели. Для Pyrus 403 при загрузке файлов он немедленно возвращает None без повторных попыток. Я также построил функцию `wrapper()`, которая является централизованным обработчиком ошибок — она перехватывает HTTPError, JSONDecodeError, ValidationError и кастомный UserError, логирует каждую appropriately и решает, пробрасывать ли дальше или вернуть None. Этот паттерн обеспечивает единообразную обработку ошибок во всех 11+ клиентах внешних сервисов.

### 11. "Расскажите о вашем опыте с asyncio"

> Весь мой текущий проект — async. Я активно использую `asyncio`: `httpx` для асинхронного HTTP, `aiokafka` для метрик, `asyncio.to_thread` через кастомную обёртку `convert_to_async()` для синхронных библиотек (LDAP, Confluence API, Vault). Prisma-агент использует LangGraph `StateGraph` для параллельной агрегации контекста. Я также использую `BackgroundTasks` в FastAPI для неблокирующей обработки вебхуков и `asyncio.create_task` для фоновых задач мониторинга. Я понимаю последствия GIL и когда использовать `multiprocessing` vs `asyncio`.

### 12. "Какой у вас опыт с наблюдаемостью?"

> Я инструментирую каждое приложение структурированным JSON-логированием, дашбордами Grafana и алертами. В summarizer-it логи идут в stdout и Graylog. Я отслеживаю RAG-метрики, такие как порог similarity score и коэффициент low-score. Я отправляю метрики в Kafka для downstream-агрегации. В моей роли SRE я строил проактивные дашборды мониторинга в Grafana, Splunk, Graylog и Kibana, которые позволяли раннее обнаружение инцидентов до воздействия на пользователей. Я понимаю, что для систем на основе агентов наблюдаемость ещё более критична — нужно отслеживать вызовы инструментов, измерять латентность по каждой интеграции и обнаруживать, когда агенты получают деградированные ответы.

### 16. "Как вы справляетесь с идемпотентностью в ваших API?"

> В summarizer-it я использую очередь задач (список на уровне модуля) для предотвращения дублирующей обработки вебхуков — хендлер засыпает на 2,5 секунды перед проверкой, затем удаляет задачи после обработки. Для обновлений Pyrus я использую `update_task`, который по своей природе идемпотентен (установка поля в значение). Для вызовов LLM я отслеживаю счётчики выполнений по каждому промпту для предотвращения бесконечных циклов. Я понимаю, что для tool API, вызываемых агентами, идемпотентность критична — агенты могут повторять попытки, и деструктивные операции требуют безопасных значений по умолчанию. Я бы реализовал idempotency-ключи или дедупликационные токены для любой операции с побочными эффектами.

### 17. "Как бы вы спроектировали новую интеграцию с инструментом с нуля?"

> Основываясь на моём опыте построения паттерна реестра промптов в summarizer-it: 1. Определить схему входных данных инструмента (Pydantic-модель) и контракт выходных данных. 2. Построить клиент с авторизацией, повторными попытками, rate limiting и маппингом ошибок. 3. Написать интеграционные тесты против upstream-сервиса. 4. Добавить структурированное логирование и метрики с первого дня. 5. Создать шаблон, чтобы следующая интеграция следовала тому же паттерну — в моём проекте каждый новый клиент наследуется от `AsyncHTTPClient` и следует той же структуре. 6. Задокументировать описание инструмента понятно — агенты рассуждают на основе этих описаний.
